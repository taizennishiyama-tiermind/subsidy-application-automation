# 補助金申請テンプレート書き込みスキル

## 概要

`README.md` を正として、テンプレート分析から書き込み完了までの流れを Claude がそのまま実行できるように整理したスキル。

### テンプレート知見の格納ルール（必須）

テンプレートの解析結果・マッピング定義は **「補助金名 × テンプレートファイル名」の組み合わせごとに専用フォルダ** に格納する。

```
grants/{補助金名}/templates/{テンプレートファイル名（拡張子なし）}/
  ├── {元のテンプレートファイル}   ← 実体ファイル
  ├── TEMPLATE_SKILL.md           ← このテンプレート専用の解析結果
  └── notes.md                    ← このテンプレート専用の補足運用メモ（任意）
```

- **他の補助金フォルダには絶対に書き込まない**
- **同じ補助金でも別テンプレートファイルなら別フォルダ**を作成する
- 公募回・バージョンが変わったら**既存フォルダを上書きせず新フォルダ**を作成する
- `/draft` 時は対象テンプレートのフォルダを明示して参照する

### 出力ファイルの格納ルール（必須）

`/fill` で生成した完成版ファイルは **`workspace/{案件名}/output/`** に保存する。
グローバルな `output/` フォルダには保存しない。

出力形式はテンプレートに合わせて選択する:
- 公式テンプレートが `.xlsx` → 完成版 `.xlsx`
- 公式テンプレートが `.docx` → 完成版 `.docx`
- 直接書き込み困難な場合 → 完成版 `.md` または `.docx`（転記箇所を明示）
- 複数形式が要求される場合 → 必要な形式をすべて `output/` に出力

## 実行原則

1. フェーズ0のテンプレート分析は省略しない
2. **全シートを必ず確認する。入力シートだけでなく、記入例、説明、見本、注意書きシートも参照する**
3. **各シートの役割を調べ、提出対象シートと参照専用シートを明確に分ける**
4. セル結合は左上セルにだけ書き込む
5. 数式セルは上書きしない
6. プレースホルダーがなくても停止しない
7. **提出対象シートの入力すべき箇所は全て特定し、埋められる情報は全て必ず埋める**
8. **提出対象シートの入力欄に例文・記入見本・ダミーテキストが入っている場合は、実値を書き込む前に削除する**
9. 自動書き込み後は必ず検証する
10. **毎回そのテンプレート自体を再解析し、前回版の固定セルだけを盲信しない**
11. **記入位置を推定した場合は、ラベルアンカー、周辺セル、結合セル、表構造を根拠として残す**
12. **必須項目の差し込み先を解決できない場合は、空欄出力で進めずに失敗として扱う**

## 標準フロー

### フェーズ0: テンプレート分析

Excel テンプレートはまず詳細分析する。

```bash
python .claude/skills/subsidy-application-filler/scripts/analyze_template.py \
  /path/to/template.xlsx \
  template_analysis.json
```

確認対象:
- 全シートの一覧と役割
- シート構成
- セル結合
- Named Range
- 動的表
- 数式セル
- データ検証セル

全シート確認のルール:
- 最初から「このシートは不要」と決めつけない
- 記入例、説明、見本シートにも入力ルール、記載順、禁止事項、計算ロジックが書かれている可能性があるため必ず読む
- 最終的に「直接入力対象」「参照専用」「自動計算」「提出不要参考」のいずれかへ分類する
- **「記入例」「記入方法」「説明」「見本」シートは参照専用であり、原則として書き込まない**
- **実際に値を書き込むのは、申請に提出するシート、またはその提出シートを成立させる元入力セルだけに限定する**
- **提出対象シートの入力欄に既存の記入例や例文が入っている場合は、参照専用情報として扱わず、書き込み前に消去する**

### フェーズ1: 情報抽出

ヒアリング内容や既存メモから構造化データを作る。

```bash
python .claude/skills/subsidy-application-filler/scripts/extract_data.py \
  /path/to/hearing_notes.txt \
  extracted_data.json
```

必要なら第3引数でカスタムスキーマを渡す。

### フェーズ2: マッピング定義作成

テンプレート分析結果と抽出データを照合し、`references/mapping_schema.md` に従ってマッピング定義を作る。

優先順位:
1. Named Range
2. 明示的なプレースホルダー
3. ラベルに隣接する入力セル
4. 結合セルの左上
5. 表構造、見出し、欄サイズ、周辺文言からの推定

追加ルール:
- 1シート目だけで判断しない。**全シート分の入力項目一覧を作ってから**マッピングを確定する
- 直接入力対象だけでなく、別シートから数式参照される元セルも特定する
- 同じ意味の項目が複数シートにある場合は、元入力セルと参照セルを区別して記録する
- 空欄のまま残る入力欄がある場合は、未入力理由を明示する
- **記入例シートや記入方法シートの空欄は埋めない**
- **提出対象に分類したシートの必須欄は空欄のまま残さない**
- **固定セルマッピングは初期候補として扱い、毎回ラベルアンカーと周辺構造で再確認する**
- **同一テンプレート名でも版が変わったらセル位置がずれる前提で、差し込み先を都度再解決する**
- **解決できたセル一覧、未解決項目、参照専用として除外したシート名を必ず記録する**

参考:
- `examples/sample_mapping.json`
- `examples/sample_word_mapping.json`

### フェーズ3: データ埋め込み

Excel への書き込みは `fill_template.py` を使う。

```bash
python .claude/skills/subsidy-application-filler/scripts/fill_template.py \
  /path/to/template.xlsx \
  my_mapping.json \
  extracted_data.json \
  /path/to/output.xlsx
```

対応する埋め込みタイプ:
- `simple_cells`
- `named_ranges`
- `dynamic_tables`

### フェーズ4: 検証

次を必ず確認する。

1. 全シートを見たうえで、**提出対象に分類した箇所だけ**が漏れなく埋まっているか
2. 数式セルが壊れていないか
3. 日付と金額のフォーマットが正しいか
4. 動的表の行ズレがないか
5. シート間参照が崩れていないか
6. 提出できる見た目か
7. 提出対象シートの入力欄にテンプレート由来の例文やダミー文が残っていないか
8. 記入例 / 記入方法 / 説明シートが書き換わっていないか
9. 必須項目の未解決がゼロか

## プレースホルダーなしテンプレートの扱い

`/add-template` や `/fill` で明示的な差し込みポイントがなくても停止しない。
次の順で入力位置を推定する。

1. 欄ラベル
2. 右隣または直下の空欄
3. 結合セルの左上
4. 同一セクションの並び
5. 表の列見出し
6. Word の場合は見出し順、表セル、欄サイズ

推定で埋めた箇所は、根拠をテンプレート解析メモまたは案件メモに残す。

推定前提の追加ルール:
- まず全シートを読み、記入例や説明シートから入力意図を把握する
- あるシートだけでは判断できない場合は、他シートの同名欄や説明文を参照して補完する
- 「入力不要」と判断する場合も、その根拠を残す
- **参照専用シートは読むが書かない。提出対象シートに反映すべき内容だけを抽出する**
- **推定は「ラベル一致 → 右隣 / 直下候補 → 結合セル左上 → 表構造」の順で行い、候補が競合したら最も根拠の強い位置だけを採用する**
- **推定結果は書き込み前に検証し、必須項目が未解決なら失敗として止める**

## 注意点

### セル結合

- 書き込み先は必ず左上セル
- 結合範囲内の他セルを直接指定しない

### 数式セル

- マッピング対象に含めない
- 合計欄は `auto_sum` などで再生成する

### 例文セル

- 提出対象シートの入力欄にある例文・記入見本・ダミーテキストは、実値を書き込む前に削除する
- ただし「記入例」「記入方法」「説明」「見本」シートそのものは参照専用であり、削除・編集しない

### フォーマット

- `date`
- `currency`
- `number`
- 必要に応じて `percentage`

## 実運用での使い方

- `/add-template` ではテンプレート解析と不足項目抽出に使う。解析結果は `grants/{補助金名}/templates/{テンプレートファイル名}/` 専用フォルダに保存する
- `/fill` では差し込み可否の判断、マッピング補正、出力生成に使う
- 出力ファイルは `workspace/{案件名}/output/` に保存する（Excel・Word・Markdown いずれも同じ場所）
- 元テンプレートへ直接書けない場合でも、完成版 `.docx`、`.xlsx`、`.md` のいずれかを生成して処理を止めない
- どのケースでも、**全シート確認 → 提出対象シートの全量特定 → 埋め込み → 提出対象中心の検証** の順を崩さない
- 補助金別 `fill_template.py` では、共通のアンカー解決ロジックを通して「今回投入された版」の差し込み先を確定してから書く
