# Grant Writer

外部の `grant-writer.skill` を、このリポジトリの補助金申請フローに合わせて採用した版。

## 使う場面

- `/draft`
- `/add-template`
- テンプレートの各設問が何を評価しているか整理したいとき
- 申請書本文と `application_data.json` を同時に整えたいとき

## 基本原則

1. 審査員は短時間で読む
2. 抽象論ではなく数値と固有名詞で書く
3. 設備説明だけで終わらず、課題から効果までつなぐ
4. テンプレートの欄サイズと設問意図に合わせて文章量を調整する

## 4つの骨格

1. 現状
2. 課題
3. 解決策
4. 効果

## パターン選択

- 課題解決型
- 成長機会型
- 変革・転換型
- 差別化・高付加価値型

詳細な書き分けは `references/story_patterns.md` を必要時に読む。

## `/draft` での流れ

1. `workspace/{案件名}` の情報を読む
2. `grants/{補助金名}/overview.md` と `docs/` を読む
3. `workspace/{案件名}/templates/{補助金名}/.../TEMPLATE_SKILL.md` があれば必ず読む
4. 最適なストーリーパターンを1つ選ぶ
5. セクション別構成案を先に示す
6. 不足情報を洗い出す
7. `draft_application.md` と `application_data.json` に落とす

## `/add-template` での流れ

1. 全シート / 全セクションを確認した解析結果を前提にする
2. 各欄がどの審査観点を見ているかを解釈する
3. ヒアリング情報のどこが対応するかを整理する
4. 充足 / 推測 / 不足を判定する
5. 不足質問を1問ずつ作る
6. 埋める値は最終的に `application_data.json` に反映する

## 設問解釈のルール

- 「現状」「課題」「背景」: 課題認識
- 「強み」「自社の特徴」: 差別化、実現可能性
- 「取組内容」「導入設備」: 解決策の妥当性
- 「効果」「目標」: 事業性、定量効果
- 「賃上げ」「給与」: 政策要件
- 「スケジュール」「実施体制」: 実現可能性

## 出力ルール

- `draft_application.md`: 読ませる本文
- `application_data.json`: 埋める値

両者に差が出ないよう、ドラフトの各段落と JSON キーの対応を意識する。

## レビュー観点

審査員視点の細かい確認は `references/review_checklist.md` を参照する。
