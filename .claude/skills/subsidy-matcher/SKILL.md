# 補助金マッチングスキル

`README.md` と `.claude/commands/match.md` を前提に、補助金候補の比較を再現性高く行うためのスキル。
この skill は「それっぽい候補を挙げる」ためではなく、**案件情報・制度要件・締切・準備負荷を同時に見て、実際に進められる候補に絞る**ために使う。

## コア目的

1. `workspace/{案件名}` の情報を基に案件条件を整理する
2. `grants/` に保存済みの制度を横断比較する
3. 必要に応じて公式情報で現在公募中の候補を追加調査する
4. 第一候補から第三候補まで、理由付きで提案する
5. 補助金確定後に申請準備ガイドへつなぐ

## 参照ファイル

- `README.md`
- `references/candidate_scoring.md`
- `references/official_sources.md`
- `references/strategy_output_template.md`
- `references/precheck_checklist.md`
- `examples/sample_match_output.md`
- `scripts/build_search_queries.py`
- `scripts/score_candidates.py`

## 標準フロー

### Phase 1: 案件の前提整理

最初に次を必ず整理する。

- 業種、事業内容
- 従業員規模
- 所在地
- 投資内容
- 投資金額
- 解決したい課題
- 実施時期
- 財務上の制約や補助金要件上の懸念

出所は次の優先順位で使う。

1. `workspace/{案件名}/hearing_notes.md`
2. `workspace/{案件名}/company_data.md`
3. 過去に同案件で更新された `strategy.md`

### Phase 2: 案件の適合条件を構造化

案件を次の観点で要約する。

- 課題カテゴリ: DX / 設備投資 / 販路開拓 / 新事業 / 省力化 / 賃上げ / 研究開発
- 申請主体: 法人 / 個人事業 / 小規模 / 中小
- 投資分類: ソフトウェア / 機械設備 / 外注 / 広報 / 店舗 / 試作開発
- 地域制約: 全国型 / 都道府県型 / 市区町村型
- スピード要件: 今すぐ必要 / 数ヶ月以内 / 次回公募待ち可

必要なら `scripts/build_search_queries.py` で検索軸を組み立てる。

### Phase 3: 保存済み補助金の一次評価

`grants/*/overview.md` を全件確認し、`docs/` があればそちらを優先する。
この段階では、まず保存済み制度を漏れなく比較する。

評価では `references/candidate_scoring.md` の 6 軸を使う。
迷う場合は `scripts/score_candidates.py` 用の JSON を作って補助的に採点する。

### Phase 4: 追加調査

保存済み候補だけで十分でない場合、または地域性・最新締切が重要な場合は追加調査を行う。

追加調査の原則:

1. `references/official_sources.md` の優先順で確認する
2. 締切、補助率、対象者要件、対象経費、事前準備を必ず公式情報で確認する
3. 民間まとめサイトだけで結論を出さない
4. 地域補助金は自治体公式ページの募集要項まで見る

### Phase 5: 実現性チェック

候補を順位付けする前に、次を確認する。

- 締切までに必要準備が間に合うか
- `gBizID` や事前登録が必要か
- 見積書、決算書、加点資料を揃えられるか
- 賃上げ、付加価値額、給与総額などの要件を現実的に達成できるか
- 補助額だけ高くても、申請負荷や採択難度が過大でないか

このチェックは `references/precheck_checklist.md` を使う。

### Phase 6: 提案

提案時は `references/strategy_output_template.md` の形式に沿う。

必ず含めるもの:

- 第一候補、第二候補、第三候補
- 各候補の補助額イメージ
- その案件で通しやすい訴求ポイント
- リスクと注意点
- 申請締切
- 公式 URL
- どんな条件なら順位が逆転するか

### Phase 7: 補助金確定後の接続

候補確定後は、以下までつなげる。

- `workspace/{案件名}/strategy.md` に記録
- 申請準備ガイドの提示
- `grants/` 未登録制度なら登録提案
- テンプレート準備の案内

## スコアリングの基本ルール

各10点、合計60点。

- 事業内容の適合性
- 規模要件
- 投資額の適合
- 要件充足度
- スケジュール
- 採択可能性

ただし、**総合点だけで結論を出さない**。以下は減点以上に強く扱う。

- 締切が極端に近い
- 必須要件を満たせない
- 投資内容が補助対象外
- 申請準備が現実的に間に合わない

このような候補は高得点でも第一候補にしない。

## 出力の質を上げるためのルール

- 金額だけでなく「この案件での想定補助額」を出す
- 制度の一般説明ではなく、その会社に当てた理由を書く
- 「通りそう」ではなく「何が評価されやすいか」を明示する
- リスクは曖昧にせず、申請前に何を確認すべきかまで書く
- 締切が近い案件では、次回公募待ちを選択肢として残す

## 保存済み制度が少ない場合

`grants/` に登録済み制度が少なくても止まらない。
保存済み 4 件だけで無理に結論を出さず、必要なら追加調査で候補を広げる。

## 注意

- 古い公募回の情報をそのまま使わない
- 公式 URL を確認してから `strategy.md` に記録する
- 保存済み `grants/` にない候補が有力なら `/add-grant` を提案する
- 「第一候補なし、第二候補から提案」の方が誠実ならそのように示してよい
