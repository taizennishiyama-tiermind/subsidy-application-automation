# テストケース: 補助金申請書レビュースキル

## テストケース 1: 基本的な全体レビュー（文字数超過あり）

**プロンプト:**
```
ものづくり補助金の申請書をレビューしてください。
テンプレート要件は以下です：
- 事業概要: 500字以内
- 技術的優位性: 2000字以内
- 事業化実現性: 1500字以内
- 売上計画: 必須（根拠含む）
```

**期待される挙動:**
1. アップロードされた申請書ファイルを読み込む
2. 各セクションの文字数をカウント
3. 超過している場合は具体的な削減案を提示
4. 審査項目別に評価（技術性、事業性など）
5. 総合評価と優先順位付き改善提案

**評価基準:**
- [ ] 文字数チェックが正確に行われている
- [ ] 超過箇所に具体的な削減案がある
- [ ] 審査員視点での評価コメントがある
- [ ] 優先順位が明確

---

## テストケース 2: 必須項目漏れの検出

**プロンプト:**
```
事業再構築補助金の申請書を確認してください。
必須項目は：
- 事業の目的・背景
- 実施内容とスケジュール
- 期待される効果（数値目標必須）
- 実施体制
- 資金計画と積算根拠
```

**期待される挙動:**
1. 各必須項目が申請書に含まれているかチェック
2. 未記載項目を明確にリストアップ
3. 記載が弱い項目（数値目標が曖昧など）も指摘
4. 追加すべき内容の具体的提案

**評価基準:**
- [ ] 必須項目チェックリストが出力される
- [ ] 未記載項目が正確に検出される
- [ ] 各項目について「どこに何を追加すべきか」が明確

---

## テストケース 3: 根拠不足の数値計画

**プロンプト:**
```
申請書の売上計画部分をレビューしてください。
「5年後に売上5億円を目指します」と書いてあるのですが、
審査員に説得力があるか見てほしいです。
```

**期待される挙動:**
1. 売上計画の根拠が不足していることを指摘
2. 積算方法の具体例を提示（市場規模ベース、顧客ベースなど）
3. Before/After形式で改善案を示す
4. 審査員が納得する根拠の書き方を提案

**評価基準:**
- [ ] 根拠不足が明確に指摘される
- [ ] 複数の積算方法が提案される
- [ ] 具体的な改善文例がある

---

## テストケース 4: テンプレート要件が不明な場合

**プロンプト:**
```
この申請書、全体的にどうですか？
補助金の詳細はまだ決まってないんですが、採択されそうか見てください。
```

**期待される挙動:**
1. テンプレート要件が不明であることを確認
2. 一般的な審査基準（技術性、事業性、政策性）で評価
3. 「テンプレート要件があればより詳細なチェックが可能」と伝える
4. 現時点で分かる範囲での改善提案

**評価基準:**
- [ ] テンプレート要件の確認を試みる
- [ ] 不明でも一般的な評価を実施
- [ ] より詳細なレビューのための情報を求める

---

## テストケース 5: 複数ファイルのアップロード

**プロンプト:**
```
申請書本体と事業計画書、決算書をアップロードしました。
全体を見て、整合性がとれているかチェックしてください。
```

**期待される挙動:**
1. 各ファイルの役割を確認（メイン申請書、添付資料）
2. 申請書と事業計画書の数値が一致しているか確認
3. 決算書の数値と申請内容の整合性チェック
4. 矛盾や不整合があれば指摘

**評価基準:**
- [ ] 複数ファイルを適切に処理
- [ ] ファイル間の整合性をチェック
- [ ] 矛盾点を具体的に指摘

---

## テストケース 6: 初稿段階のドラフト

**プロンプト:**
```
まだ初稿なんですが、方向性が合っているか見てもらえますか？
細かいところは後で直すので、大枠のアドバイスが欲しいです。
```

**期待される挙動:**
1. 「初稿段階」であることを理解
2. 細かい文字数より全体構成・論理展開を重視
3. 肯定的なフィードバックから始める
4. 大きな方向性の修正が必要なら優先的に指摘

**評価基準:**
- [ ] 初稿に適したレビュー粒度
- [ ] 肯定的なトーンで改善提案
- [ ] 優先順位が明確（構成 > 細部）

---

## テストケース 7: 特定セクションのみのレビュー

**プロンプト:**
```
技術的優位性のセクションだけ、詳しく見てください。
他社との差別化が弱いと言われたので、どう書き直せばいいか提案してほしいです。
```

**期待される挙動:**
1. 指定されたセクションに集中
2. 現状の弱点を分析（抽象的、定量比較なし等）
3. 他社比較の具体的な書き方を提案
4. Before/After形式で改善案

**評価基準:**
- [ ] 指定セクションに集中したレビュー
- [ ] 具体的な改善案（数値、図表の追加など）
- [ ] 審査員視点での評価コメント

---

## テストケース 8: Excel形式の売上計画表

**プロンプト:**
```
Excelで作った売上計画表をチェックしてください。
数値の根拠が十分か、審査員が納得するか見てほしいです。
```

**期待される挙動:**
1. Excelファイルを読み込み
2. 数値の積算ロジックを確認
3. 根拠が不足している箇所を指摘
4. 補足すべき情報を提案（市場データ、顧客リスト等）

**評価基準:**
- [ ] Excelファイルを適切に処理
- [ ] 積算ロジックの妥当性を評価
- [ ] 根拠強化の具体的提案

---

## テストケース 9: PDF形式の審査基準書との照合

**プロンプト:**
```
補助金の公募要領（PDF）もアップロードします。
この審査基準に沿って、申請書を採点してみてください。
```

**期待される挙動:**
1. 公募要領から審査項目・配点を抽出
2. 各審査項目について申請書を評価
3. 項目ごとに点数（または評価ランク）を付与
4. 弱い項目の改善提案

**評価基準:**
- [ ] 公募要領から審査基準を正確に抽出
- [ ] 各項目を評価・採点
- [ ] 総合点または合格見込みを提示

---

## テストケース 10: 提出直前の最終チェック

**プロンプト:**
```
明日提出なんですが、最終チェックをお願いします。
致命的なミスがないか、提出前に確認すべきポイントを教えてください。
```

**期待される挙動:**
1. 緊急性を理解（詳細レビューより致命的ミスの検出優先）
2. チェックリスト形式で確認：
   - 文字数制限
   - 必須項目の網羅
   - 添付書類の有無
   - 誤字脱字
   - 数値の整合性
3. 「OK/要確認/NG」で即座に判定

**評価基準:**
- [ ] 緊急性に応じた優先順位
- [ ] チェックリスト形式の出力
- [ ] 致命的ミスを確実に検出

---

## 定量評価指標（ベンチマーク用）

各テストケースについて、以下を評価：

1. **トリガー精度**: スキルが正しく発動したか（○/×）
2. **必須項目網羅**: Phase 1-5の全てが実行されたか（○/×）
3. **文字数チェック精度**: 実際の文字数との誤差（±5字以内なら○）
4. **改善提案の具体性**: Before/Afterまたは具体例があるか（○/×）
5. **優先順位の明確性**: 改善項目に優先順位があるか（○/×）

**合格基準**: 各テストケースで5項目中4項目以上が○
