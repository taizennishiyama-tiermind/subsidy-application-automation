# 補助金申請書レビュー・改善スキル

## 概要

`README.md` を正として、`/review` で行うべきレビュー手順をそのまま実行できるよう整理したスキル。
審査員視点のレビューに加えて、テンプレート整合性チェックと優先順位付きの改善提案を必須とする。

## コア機能

1. 審査基準に基づくレビュー
2. テンプレート整合性チェック
3. Before/After を含む改善提案
4. 総合評価と優先順位付け

## 標準フロー

### Phase 1: 状況把握

最初に次を確認する。

1. 申請書ドラフトの取得
2. 対象補助金名の確認
3. テンプレート要件ファイル、または文字数制限・必須項目・審査項目の確認
4. 全体レビューか、特定セクションレビューかの確認

扱う入力形式:
- `draft_application.md`
- PDF
- Word
- Excel
- テキスト形式の申請書

### Phase 2: テンプレート整合性チェック

このフェーズは必ず実行する。

#### 2.1 文字数チェック

各セクションについて以下を整理する。

```text
【セクション名】 現在: XXX字 / 制限: YYY字
状態: OK / 超過 / 不足
```

超過時は削減が必要な文字数と、削減候補を具体的に示す。

#### 2.2 必須項目チェック

最低限、次のような観点で抜け漏れを確認する。

- 事業の目的・背景
- 実施内容・スケジュール
- 期待される効果・数値目標
- 実施体制・担当者
- 資金計画・積算根拠

未記載があれば、「どこに何を足すか」まで提案する。

#### 2.3 様式・体裁チェック

- 図表番号の連番
- 必須添付資料の言及
- ページ数制限
- テンプレートとの欄対応のズレ

整合性チェックの出力は次の3段で整理する。

- 適合項目
- 要修正項目
- 致命的不備

### Phase 3: 審査員視点レビュー

以下の4観点で評価する。

- 形式
- 内容
- 数値
- 審査基準対応

各観点で、次を明示する。

- 強み
- 弱み
- 減点リスク
- 改善方向

特に重視する論点:

- 技術的優位性・革新性
- 事業化実現性
- 政策適合性
- 費用対効果
- 全体の論理展開

### Phase 4: 具体的改善提案

できる限り Before/After 形式で改善案を示す。

優先して出す提案:

1. 文字数削減案
2. 数値根拠の補強案
3. 表現の具体化
4. 不足セクションの追記案

改善提案は抽象論で終わらせず、差し替え文か追加観点まで出す。

### Phase 5: 総合評価

最後に次を返す。

- 総合評価
- 改善必須項目（CRITICAL）
- 改善推奨項目（RECOMMENDED）
- 良い点
- 添付書類チェックリスト

## 運用ルール

- テンプレート整合性チェックを省略しない
- 部分レビュー依頼でも致命的不備があれば先に指摘する
- `README.md` の出力構成を優先する
